# unseen-gate

## What is this machine

Nobody is entirely sure who set it up or when. It appeared on the city network
at some point during the third Igor administration, quietly running SSH, and
everyone assumed it belonged to someone else's department.

It has clearly had occupants. The Python environment in `/opt/attacker-env`
contains a collection of libraries that have no obvious legitimate use on a
utility network. The `~/loot/` directories in several accounts hold notes from
what appears to be a previous engagement — reconnaissance fragments, partial
host lists, the kind of thing you're not supposed to leave lying around on
shared infrastructure.

Someone named Ponder seems to have been here. He left notes.

The UU P&L network lies beyond. Somewhere in it, things are running that
probably should not be running the way they are running.

## Accounts

Five accounts, all key-authenticated. Each issued to one participant before
the exercise begins. They share the same machine and can see each other's
activity if they look.

| Username    | Character               | Notes                                                      |
|-------------|-------------------------|------------------------------------------------------------|
| `ponder`    | Ponder Stibbons         | Research wizard. Built Hex. Left notes everywhere.         |
| `ridcully`  | Archchancellor Ridcully | Senior. Loud. Probably here by accident.                   |
| `librarian` | The Librarian           | Ook.                                                       |
| `dean`      | The Dean                | Excitable. Has already run nmap twice.                     |
| `hex`       | (named by Ponder)       | He named an account after his computer. Of course he did.  |

All five have identical access. No sudo. No root. No password fallback.

## Network position

```
participant
  │ SSH (port 2222 locally / port 22 on Hetzner)
  ▼
unseen-gate  10.10.0.5  (ics_internet only — not dual-homed)
```

From `unseen-gate`, participants are on `10.10.0.0/24`. They cannot reach the
enterprise, operational, or control zones directly — the firewall drops it.
They have to find another way in.

## Container behaviour

Exposes one service: SSH on port 22, key-only auth, no password fallback.

At startup, `entrypoint.sh` reads `/run/adversary-keys` (bind-mounted at
runtime) and distributes public keys to each account's `authorized_keys`.
It also copies `/run/adversary-readme.txt` to each home directory as `README`,
and plants `~/loot/prior-recon.txt` in each account.

Neither file is part of the image. Both are generated at deploy time.

## Operator setup

```bash
# Local dev — ./ctl up handles this automatically:
# adds your ~/.ssh/id_*.pub for user 'ponder'

# For a full deployment, populate before starting:
cp infrastructure/attacker-machine/adversary-keys.example \
   infrastructure/attacker-machine/adversary-keys
# Edit: one line per participant — username pubkey [comment]
# Valid usernames: ponder  hex  ridcully  librarian  dean
```

Then:
```bash
./ctl up          # generates, builds, starts — prints SSH command
./ctl firewall    # apply inter-zone rules (sudo)
```
