# Stunnel TLS gateway

`uupl-stunnel-gw` is the mTLS termination point between the SCADA system and
the turbine PLC. It accepts TLS connections on port 8502, verifies the client
certificate, and forwards plain Modbus TCP to the PLC at port 502. The intent is
mutual authentication: only the SCADA system holds a valid client certificate,
so only the SCADA system can reach the PLC via this gateway.

The TLS version is pinned to TLSv1. Upgrading to TLSv1.2 broke compatibility
with the SCADA client library in 2019, and revisiting it was deferred to the next
outage window. The server certificate expired and renewal has not been completed
(HEX-4421, won't-fix). The SCADA client was reconfigured to skip server
certificate verification rather than miss a maintenance window.

The gateway is correctly designed. The client has been reconfigured to ignore
what the gateway is saying. This is the realistic part.

## The real-world degradation pattern

TLS gateways between SCADA and PLCs are a genuine security control in modern OT
deployments. The degradation pattern here is authentic: a certificate expires,
operational pressure prevents a clean renewal, the quickest fix is to disable
verification on the client side, and the control remains in place but no longer
provides the protection it was designed for. The client now establishes TLS
without verifying who it is talking to, which means the mTLS protection is
one-sided.

## Container details

Base image: `dweomer/stunnel`. Minimal image with stunnel pre-installed.

Exposed port: 8502/tcp (TLS listener).

Certificates are volume-mounted at `/run/stunnel/` from the `certs/` directory
generated by the orchestrator. The config template (`stunnel.conf`) is also
volume-mounted from this component directory; `entrypoint.sh` substitutes
`FORWARD_TARGET` from the environment before starting stunnel.

Configuration summary:
- `sslVersion = TLSv1` (pinned, cannot be negotiated up)
- `verify = 2` (requires client certificate against CA)
- `cert` / `key`: server identity
- `CAfile`: CA certificate for client verification
- Forward target: default `10.10.3.21:502` (set via `FORWARD_TARGET` env var)

The server key is `chmod 600` at start.

## Connections

- `ics_operational`: 10.10.2.50 (the operational zone listening address)
- Forwards to PLC at 10.10.3.21:502 on `ics_control`
- Accepts connections from `distribution-scada` (10.10.2.20) SCADA system

## Protocols

TLS (TLSv1): port 8502 inbound.
Modbus TCP: port 502 outbound (plain, forwarded to PLC).

## Built-in vulnerabilities

TLSv1 pinned: TLSv1 is deprecated (RFC 8996, 2021) and disabled by default in
modern TLS libraries. The protocol downgrade was accepted in 2019. The gateway
will not upgrade regardless of what the client proposes.

Server certificate not verified by client: the SCADA stunnel client has
`verify = 0` (see the scada-lts component README). A network attacker with a
route to port 8502 could present any certificate and the client would accept the
connection.

Client certificate verified by gateway: the gateway does enforce `verify = 2`.
Only a client that presents a certificate signed by the gateway's CA will be
accepted. The client certificate and key are on the SCADA container at
`/run/stunnel-certs/client.key` (world-readable, HEX-5103).

Effectively: the mTLS is client-to-gateway only (the gateway authenticates the
client, but not vice versa from the client's perspective).

## Modifying vulnerabilities

To enable TLSv1.2: remove the `sslVersion = TLSv1` line from `stunnel.conf`.
The client configuration in `scada-lts` also requires updating if the CA is
enforced end-to-end.

To disable mTLS (remove client cert requirement): change `verify = 2` to
`verify = 0` in `stunnel.conf`. This reduces the gateway to a plain TLS
terminator.

To use a different forward target: set the `FORWARD_TARGET` environment variable
in `ctf-config.yaml` under the `stunnel_gateway` device.

To rotate certificates: replace the files in the `certs/` directory and restart
the gateway and SCADA containers.

## Hardening suggestions

Upgrade to TLSv1.2 at minimum (preferably TLSv1.3). Renew the server
certificate and re-enable `verify = 2` on the SCADA client. Restrict which
source IPs can reach port 8502 via firewall rules; only the SCADA server IP
needs access. Ensure the client key on the SCADA container is not world-readable.

## Observability and debugging

```bash
docker logs stunnel-gateway
docker exec -it stunnel-gateway sh
```

Test TLS connectivity from the SCADA container (TLSv1 client):
```bash
openssl s_client -connect 10.10.2.50:8502 -tls1 -cert client.crt -key client.key
```

The gateway logs accepted and rejected connections to stdout.

## Concrete attack paths

The gateway itself is not a direct exploitation target. Its relevance to attack
chains is:

1. The SCADA container's world-readable client key (`/run/stunnel-certs/client.key`)
   is the credential for this gateway. An attacker with a shell on the SCADA
   container can exfiltrate the key.
2. With the client certificate and key, the attacker can connect to port 8502
   from any host that reaches the operational network and issue Modbus commands
   to the PLC.
3. Because the SCADA client does not verify the gateway certificate, a
   machine-in-the-middle between the SCADA server and port 8502 can intercept
   and modify Modbus traffic without being detected.

## Before you dig in

The gateway is on `ics_operational` (10.10.2.50), not `ics_control`. It bridges
the two zones: the SCADA server connects to it on the operational side, and it
forwards to the PLC on the control side. Without the gateway, the SCADA server
has no direct path to the PLC.

The `FORWARD_TARGET` defaults to `10.10.3.21:502`. If the PLC container address
changes, the gateway needs to be rebuilt or the environment variable updated.

The stunnel config template is at `zones/control/components/stunnel-gateway/stunnel.conf`
in the repository. The version in the running container is written to
`/etc/stunnel/stunnel.conf` by `entrypoint.sh` after `FORWARD_TARGET`
substitution.

## Bottom line

TLS-to-Modbus gateway between SCADA and PLC. mTLS is technically in place but
one-sided: the gateway verifies the client certificate, but the SCADA client
does not verify the gateway. TLSv1 only. Server cert expired. The client
certificate and key are world-readable on the SCADA container, which is the
primary loot here.
