# Multi-domain configuration

## The core

The orchestrator, iptables generator, and Docker infrastructure are domain-agnostic. They take a YAML configuration and produce a working multi-zone network with firewall rules. Nothing in the core assumes turbines, power grids, or any specific OT domain.

Container implementations in `zones/` are reusable across scenarios. A Modbus RTU, an OPC-UA server, or an IEC 104 slave each gets added once and becomes available to any config that references it. The UU Power and Light scenario is the first; future scenarios reuse the same container library.

## Adding a new domain

A new domain scenario needs:

1. A configuration directory under `configs/` with its own `ctf-config.yaml`
2. Any new container implementations that do not already exist in `zones/`

The shared `zones/` library grows as new container types are added. Existing zone components (PLCs, historians, relay IEDs, actuators) are available to any config that references them.

## Two-SCADA design

The UU P&L scenario runs two Scada-LTS instances that model the Purdue Model boundary between supervisory control (L2) and operations management (L3).

Control SCADA (`uupl-hmi`, 10.10.3.10, ics_control): operators use this to view and change the physical process. It talks to field devices via Modbus, routed through the stunnel TLS gateway before it reaches the PLC. Mistakes here move metal. Default credentials `admin/admin`; never changed.

Operations SCADA (`distribution-scada`, 10.10.2.20, ics_operational): management and engineering use this to observe trends, generate reports, and optimise schedules. It talks to the historian REST API, read-only aggregated data. Mistakes here produce bad reports. Default credentials `admin/admin`; never changed for the same reason as above.

Both instances use `scadalts/scadalts` (Mango Automation base) with the same intentional misconfigurations: default credentials, Groovy script execution endpoint, and a world-readable stunnel client key left in place since a 2020 monitoring integration that was never cleaned up (ticket HEX-5103, OPEN).

The stunnel gateway (`uupl-modbus-gw`) is dual-homed across both zones. Its role is to terminate mTLS from the control SCADA and proxy plain Modbus to the PLC, a realistic pattern for retrofitting TLS onto legacy field devices. The CTF attack path: compromise control Scada-LTS → steal `client.key` (chmod 644) → talk directly to the gateway → send arbitrary Modbus writes the SCADA UI would never expose.

## Consequences for CTF documentation

Each domain scenario has its own configuration directory. Challenge runbooks reference the config by directory name rather than assuming a single `ctf-config.yaml`.

The orchestrator currently expects a single `ctf-config.yaml` path. When a second domain config is added, select it with:

```bash
CONFIG=orchestrator/configs/smart-grid.yaml ./ctl up
```

The `CONFIG=` environment variable is already supported; the per-domain config directories are the part still ahead.
