# Architecture: multi-domain configuration

## The core

The orchestrator, iptables generator, and Docker infrastructure are domain-agnostic. They take
a YAML configuration and produce a working multi-zone network with firewall rules. Nothing in
the core assumes turbines, power grids, or any specific OT domain.

The physics engines (TurbinePhysics, GridPhysics, ReactorPhysics, HVACPhysics, PowerFlow) are
domain-specific modules, but they are pluggable. The core does not depend on any one of them.

## Adding a new domain

A new domain scenario requires:

1. A configuration directory under `configs/` with its own `ctf-config.yaml`
2. Any new container implementations that do not already exist in `zones/`
3. A physics engine module if the domain has a simulated physical process

The shared `zones/` library grows over time as new container types are added. A DLMS meter, an
IEC 61850 IED, or an infusion pump controller each gets added once and becomes available to any
config that references it.

## Directory layout

```
ics-simlab/
  orchestrator/          core: config parser, Compose and firewall generator
  zones/                 shared container library
    internet/            attacker machine, admin-home
    enterprise/          legacy workstation, enterprise workstation
    operational/         historian, scada-lts (ops), engineering workstation,
                         scada-server (unwired, preserved)
    control/             scada-lts-ctrl (control HMI), turbine PLC, IED relays,
                         actuators, meter, Mosquitto broker, stunnel-gateway
    dmz/                 umatiGateway, Neuron gateway, MQTT broker, OPC-UA server,
                         IEC-104 RTU, SSH bastion, SFTP drop, NTP, DNS, syslog relay
  certs/                 TLS certs — generated by generate.py, gitignored
  configs/               (planned) per-scenario config directories
    power-ctf/           UU Power and Light scenario (current config is at root)
    smartgrid/           IEC 61850 substation scenario (future)
    healthcare/          healthcare OT scenario (future)
  infrastructure/        host-level setup (network compose, firewall)
```

## Two-SCADA design

The UU P&L scenario runs two Scada-LTS instances that model the Purdue Model
boundary between supervisory control (L2) and operations management (L3).

**Control SCADA** (`uupl-hmi`, 10.10.3.10, ics_control)
Operators use this to view and change the physical process. It talks to field
devices via Modbus — routed through the stunnel TLS gateway before it reaches
the PLC. Mistakes here move metal. Default creds `admin/admin`; never changed.

**Operations SCADA** (`distribution-scada`, 10.10.2.20, ics_operational)
Management and engineering use this to observe trends, generate reports, and
optimise schedules. It talks to the historian REST API — read-only aggregated
data. Mistakes here produce bad reports. Default creds `admin/admin`; never
changed for the same reason as above.

Both instances use `scadalts/scadalts` (Mango Automation base) with the same
intentional misconfigurations: default credentials, Groovy script execution
endpoint, and a world-readable stunnel client key left in place since a 2020
monitoring integration that was never cleaned up (ticket HEX-5103, OPEN).

The stunnel gateway (`uupl-modbus-gw`) is dual-homed across both zones. Its
role is to terminate mTLS from the control SCADA and proxy plain Modbus to the
PLC — a realistic pattern for retrofitting TLS onto legacy field devices. The
CTF attack path: compromise control Scada-LTS → steal `client.key` (chmod 644)
→ talk directly to the gateway → send arbitrary Modbus writes the SCADA UI
would never expose.

## Why not branches

Branches are for work that converges. A `smartgrid` branch and a `main` branch modelling a
power plant diverge from day one and never merge cleanly. Core improvements to the orchestrator
would need to be applied to every branch separately, or cherry-picked, which is maintenance
overhead with no benefit.

## Why not forks

A fork per domain means three copies of the orchestrator, three copies of the firewall
generator, three copies of every fix. The domains share the core; forks discard that sharing.

A fork is appropriate only if a domain scenario needs to be handed to a third party who cannot
see other configs or the full container library. That is an access-control decision, not an
architecture decision, and it does not change the recommended structure for internal development.

## Consequences for CTF documentation

Each domain scenario has its own configuration directory. Challenge runbooks reference the
config by directory name rather than assuming a single `ctf-config.yaml`.

The orchestrator and Makefile currently expect a single `ctf-config.yaml` path. When a second
domain config is created, the first change needed is a `CONFIG=` parameter on `make generate`
so that each domain can be built independently from its own config directory.
